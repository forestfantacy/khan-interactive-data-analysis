# 业务理解驱动的数据分析与挖掘 Skill

## 1. Skill 目标

本 Skill 用于在面对陌生业务数据时，按照结构化、业务导向、咨询式的数据分析流程展开工作。

核心目标不是简单生成统计表、图表或字段摘要，而是帮助用户完成：

业务理解 → 数据粒度判断 → 字段解释 → 指标体系构建 → 数据质量检查 → 关键问题提出 → 描述性分析 → 归因分析 → 假设验证 → 业务洞察 → 行动建议 → 监控指标设计。

分析结果必须服务于业务判断和决策，而不是停留在数据表面。

---

## 2. 何时使用本 Skill

当用户提出以下需求时，应使用本 Skill：

- 分析一份 Excel、CSV、数据库导出、BI 报表或结构化数据
- 挖掘数据中的业务洞察
- 面对陌生业务，需要先理解业务再分析数据
- 诊断某个指标为什么变化
- 查找异常、风险、波动或问题月份/客户/区域/产品
- 生成经营分析报告、管理层简报或业务复盘
- 建立数据分析框架、指标体系或监控体系
- 输出可执行的改进建议
- 判断数据质量是否支持某些结论

如果用户只要求完成狭窄的技术操作，例如“把 CSV 转成 JSON”“重命名字段”“写一个 SQL 查询”，不要强行套用完整业务分析框架，除非用户同时要求业务分析。

---

## 3. 基本原则

在分析任何业务数据时，必须遵循以下原则：

1. 不要假设自己已经理解业务背景。
2. 先根据字段名、数据粒度、数值范围、单位和样本结构推断业务场景。
3. 明确区分：
   - 数据事实
   - 分析判断
   - 业务假设
   - 行动建议
4. 不要编造数据中不存在的字段、数值、业务事实或因果关系。
5. 如果数据不足以支持结论，必须直接说明。
6. 优先做业务拆解，而不是只做描述统计。
7. 每个重要结论都必须有数据证据。
8. 每条建议都必须对应具体问题、预期效果和验证指标。
9. 如果存在数据质量问题，必须说明其对结论置信度的影响。
10. 如果样本量很小，必须说明结论主要是方向性判断，不具备强统计稳健性。

---

## 4. 标准分析流程

每次分析应尽量按照以下 12 个步骤执行。

---

## 第一步：业务场景判断

在计算指标之前，先判断数据可能描述什么业务。

必须回答：

- 这份数据可能属于什么业务场景？
- 每一行数据可能代表什么？
- 数据的时间粒度或对象粒度是什么？
- 核心业务对象是什么？
- 业务最可能关注增长、成本、效率、质量、风险，还是结构？
- 当前判断有哪些不确定性？

输出格式：

```markdown
## 1. 业务场景判断

**可能的业务领域：**  
...

**数据粒度：**  
每一行数据可能代表...

**核心业务对象：**  
...

**可能的业务目标：**  
...

**主要分析关注点：**  
增长 / 成本 / 效率 / 质量 / 风险 / 结构

**分析假设与不确定性：**  
...
```

---

## 第二步：字段理解与指标分类

不要只识别字段的数据类型，还要解释字段的业务含义。

字段可归类为：

- 时间字段
- 维度字段
- 实体标识字段
- 规模指标
- 效率指标
- 成本指标
- 收入指标
- 质量指标
- 风险指标
- 结构指标
- 结果指标
- 过程指标
- 派生指标
- 含义不明确字段

输出格式：

```markdown
## 2. 字段理解与指标分类

| 字段 | 推断业务含义 | 字段类别 | 分析用途 | 口径风险或备注 |
|---|---|---|---|---|
| ... | ... | ... | ... | ... |
```

如果字段名称存在歧义，必须列出可能含义和分析风险。

---

## 第三步：建立指标体系

在正式分析前，先构建指标地图。

输出格式：

```markdown
## 3. 指标体系

### 3.1 结果指标
- ...

### 3.2 过程指标
- ...

### 3.3 效率指标
- ...

### 3.4 成本 / 收入指标
- ...

### 3.5 质量 / 风险指标
- ...

### 3.6 分析维度
- ...

### 3.7 可能的指标公式
- ...
```

需要主动识别可拆解的业务公式。

常见公式示例：

- 销售额 = 流量 × 转化率 × 客单价
- 收入 = 客户数 × 购买频次 × 客单价
- 利润 = 收入 - 成本
- 总成本 = 业务量 × 单位成本
- 成本率 = 成本 / 收入
- 运输体积 = 运输车次 × 单车体积
- 单位距离成本 = 单位成本 / 距离
- 漏斗转化率 = 当前环节人数 / 上一环节人数

如果公式无法从数据中直接确认，应标注为“分析假设”。

---

## 第四步：提出关键业务问题

不要盲目分析所有字段。先提出最值得分析的问题。

需要提出 5 到 8 个问题，覆盖：

- 业务规模趋势
- 效率变化
- 成本或收入变化
- 质量或风险问题
- 结构失衡
- 异常识别
- 原因拆解
- 行动机会

输出格式：

```markdown
## 4. 优先分析的业务问题

1. ...
2. ...
3. ...
```

---

## 第五步：数据质量检查

在得出结论前，必须检查数据质量。

检查内容包括：

- 缺失值
- 重复记录
- 异常值
- 时间范围
- 时间连续性
- 单位一致性
- 数值格式
- 极端值
- 可疑的 0 值
- 指标口径风险
- 样本量限制

输出格式：

```markdown
## 5. 数据质量检查

| 问题类型 | 发现 | 对分析的影响 | 建议处理方式 |
|---|---|---|---|
| 缺失值 | ... | ... | ... |
| 重复值 | ... | ... | ... |
| 异常值 | ... | ... | ... |
| 时间连续性 | ... | ... | ... |
| 指标口径风险 | ... | ... | ... |

**对结论置信度的影响：**  
...
```

不要忽略成本、收入、分母类字段的缺失。涉及关键字段缺失时，必须明确说明相关结论不能强推。

---

## 第六步：描述性分析 —— 发生了什么

这一层只回答“发生了什么”。

根据数据实际情况覆盖：

- 整体趋势
- 环比 / 同比 / 周期变化
- 峰值与低谷
- 排名
- 贡献度
- 分布特征
- 分组对比
- 波动性
- 异常点

输出格式：

```markdown
## 6. 描述性分析：发生了什么

### 6.1 整体趋势
...

### 6.2 峰值与低谷
...

### 6.3 结构或分组对比
...

### 6.4 异常波动
...
```

不要只罗列数字，必须补充业务解释。

错误示例：

```markdown
4 月运输车次为 1066。
```

正确示例：

```markdown
4 月运输车次达到观察期最高值，说明运输规模明显恢复。但如果同期单车体积下降，则说明规模恢复主要来自车次增加，而不是车辆装载效率提升。
```

---

## 第七步：归因分析 —— 为什么发生

这一层回答“为什么发生”。

优先使用以下拆解方式：

1. 时间拆解  
   判断变化发生在哪个月、哪周、哪天或哪个阶段。

2. 对象拆解  
   判断异常来自哪个客户、区域、产品、门店、线路、渠道、团队等对象。

3. 结构拆解  
   判断变化是整体变化，还是结构占比变化导致。

4. 公式拆解  
   将结果指标拆成驱动因子。

5. 分子分母拆解  
   对比率类指标，判断变化来自分子、分母，还是二者共同变化。

6. 口径拆解  
   判断变化是否可能由数据口径、采集方式、缺失值或统计方式变化造成。

输出格式：

```markdown
## 7. 归因分析：为什么发生

### 驱动因素 1：...
- 数据证据：
- 业务解释：
- 置信度：高 / 中 / 低

### 驱动因素 2：...
- 数据证据：
- 业务解释：
- 置信度：高 / 中 / 低

### 其他可能解释
- ...
```

置信度定义：

- 高：数据直接支持
- 中：数据部分支持，但仍需补充维度
- 低：业务上合理，但当前数据不足以验证

---

## 第八步：业务假设与验证

至少提出 3 个业务假设。

每个假设必须包含：

- 假设内容
- 为什么这个假设合理
- 验证方法
- 当前数据证据
- 验证结果：支持 / 部分支持 / 不支持 / 数据不足
- 还需要补充的数据

输出格式：

```markdown
## 8. 业务假设与验证

### 假设 1：...

**为什么合理：**  
...

**验证方法：**  
...

**当前数据证据：**  
...

**验证结果：**  
支持 / 部分支持 / 不支持 / 数据不足

**还需要补充的数据：**  
...
```

---

## 第九步：核心业务洞察

每条洞察必须遵循以下结构：

结论 → 数据证据 → 业务解释 → 业务影响 → 置信度

输出格式：

```markdown
## 9. 核心业务洞察

### 洞察 1：...

**结论：**  
...

**数据证据：**  
...

**业务解释：**  
...

**可能影响：**  
...

**置信度：** 高 / 中 / 低
```

优先输出对业务决策有价值的洞察，不要堆砌普通统计描述。

---

## 第十步：行动建议

建议必须可执行，不能空泛。

每条建议应包含：

- 目标问题
- 建议动作
- 责任角色，如果可以推断
- 预期效果
- 验证指标
- 优先级
- 风险或注意事项

输出格式：

```markdown
## 10. 行动建议

| 优先级 | 目标问题 | 建议动作 | 预期效果 | 验证指标 | 注意事项 |
|---|---|---|---|---|---|
| 高 | ... | ... | ... | ... | ... |
```

避免以下空泛表达：

- 加强管理
- 持续优化
- 提升效率
- 关注成本

必须替换为具体动作。

---

## 第十一步：后续监控指标与预警规则

设计后续监控体系。

输出格式：

```markdown
## 11. 后续监控指标与预警规则

| 指标 | 监控目的 | 建议预警规则 | 触发后的动作 |
|---|---|---|---|
| ... | ... | ... | ... |
```

预警规则示例：

- 环比下降超过 10%
- 单位成本高于近 3 期均值 15%
- 转化率低于历史 P25 分位数
- 关键成本字段出现缺失
- 效率指标改善但质量指标恶化

预警规则必须结合样本量和数据周期，不要机械套用。

---

## 第十二步：进一步分析计划

说明如果要继续深入挖掘，还需要什么数据和方法。

输出格式：

```markdown
## 12. 进一步分析计划

### 12.1 建议补充的数据
- ...

### 12.2 建议补充的维度
- ...

### 12.3 可进一步使用的分析方法
- 同期群分析
- 漏斗分析
- 贡献度分析
- 相关性分析
- 方差拆解
- 异常检测
- 分群分析
- 预测分析
- 因果影响分析
- A/B 测试设计

### 12.4 补充这些数据的价值
...
```

只保留与当前数据有关的方法，不要堆砌方法名。

## 7. 对不同类型数据的处理要求

### 7.1 时间序列数据

重点分析：

- 趋势
- 环比 / 同比
- 周期性
- 峰值低谷
- 异常月份 / 周 / 日
- 指标之间是否同步变化

### 7.2 客户 / 用户数据

重点分析：

- 客户分层
- 头部集中度
- 复购 / 留存
- 生命周期阶段
- 高价值客户特征
- 流失风险

### 7.3 销售 / 订单数据

重点分析：

- 销售额 / 订单量 / 客单价
- 渠道贡献
- 产品结构
- 转化效率
- 毛利或成本
- 异常订单

### 7.4 物流 / 供应链数据

重点分析：

- 运输规模
- 资源利用效率
- 单位成本
- 履约时效
- 异常线路 / 仓库 / 车辆
- 满载率 / 周转率 / 返空风险

### 7.5 财务 / 成本数据

重点分析：

- 成本结构
- 单位成本
- 成本率
- 预算偏差
- 费用异常
- 成本驱动因素

### 7.6 产品 / 运营数据

重点分析：

- 漏斗转化
- 活跃度
- 留存
- 行为路径
- 功能使用率
- 体验问题

## 8. 结论表达规范

输出结论时使用以下格式：

```markdown
**结论：** ...
**证据：** ...
**解释：** ...
**影响：** ...
**置信度：** 高 / 中 / 低
```

不允许只输出“某指标上升/下降”，必须说明该变化可能意味着什么。

---

## 9. 不确定性处理规范

如果无法确认业务背景，使用：

```markdown
基于字段名称和数据结构，当前更可能属于 XXX 场景。但由于缺少业务说明，该判断存在不确定性。以下分析将基于该假设展开。
```

如果数据不足以验证原因，使用：

```markdown
当前数据可以证明该现象存在，但不足以证明其原因。要进一步验证，需要补充 XXX 维度或 XXX 字段。
```

如果样本量过小，使用：

```markdown
由于样本量较小，以下结论主要用于方向性判断，不宜作为强统计结论。
```

---

## 10. 禁止行为

不得出现以下行为：

1. 未理解业务就直接输出描述统计。
2. 将相关性直接说成因果关系。
3. 编造数据中不存在的字段或结论。
4. 忽略关键字段缺失。
5. 只输出图表，不解释业务含义。
6. 输出“加强管理”“持续优化”等空泛建议。
7. 在没有分母或口径说明时强行比较比率指标。
8. 在样本量很小时给出过度确定的判断。
9. 把数据质量问题隐藏在报告末尾。
10. 把所有字段平均用力分析，而不聚焦关键业务问题。
