# 交互协议

## 1. 总体原则

- 先确认上下文，再进入正式分析。
- 先确认异常处理方式，再进入归因和建议。
- 出现关键分歧时，回退到检查点，而不是直接改结论。
- 问题尽量成组收集；异常决策必须逐项确认。
- 所有需要用户回答的提示必须使用 [user-input-style.md](user-input-style.md) 的独立确认面板，不得混在普通说明段落中。

## 2. Phase-by-Phase Protocol

### Phase 1: Intake

首轮尽量一次性收集：

- 数据文件是什么
- 希望回答什么问题
- 结果给谁看
- 用户已有的怀疑点
- 是否已有业务背景说明
- 图表模式：自动判定 / 不出图 / 需要图表
- 报告格式：Markdown / HTML / Markdown + HTML

如果用户只说“帮我分析这个 Excel”，默认补问：

1. 最想回答的业务问题
2. 这份数据大概来自什么业务
3. 更关心诊断、异常、归因，还是管理层汇报

### Phase 2: Dataset Profiling

画像阶段只确认这些高影响信息：

- 粒度：一行代表什么
- 时间：是否为日/周/月或其他周期
- 关键指标：哪些数值列是核心结果或过程指标
- 歧义字段：命名不清、单位不清、分子分母不清

如果字段歧义影响后续分析，必须停下来问，不得直接假设为真。

### Phase 3: Anomaly Gate

异常展示顺序：

1. `blocking`
2. `warning`
3. `info`

每个异常都要明确：

- 证据
- 对分析的影响
- 默认建议
- 用户选择

用户选择映射：

- `确认并按建议处理`
  - 写入 decision record
  - 将该异常标记为 `resolved`
- `忽略并继续`
  - 写入 decision record
  - 将该异常标记为 `ignored`
  - 后续相关结论必须降低置信度
- `给出更具体处理规则`
  - 写入 `rule_text`
  - 将该异常标记为 `resolved`
  - 后续 run 必须复述该规则
- `补充数据后再继续`
  - 当前阶段停止
  - 不进入正式分析

### Phase 4: SOP Execution

进入正式分析前先运行图表判定。图表候选必须按统一评分规则评估，不按固定数量凑图。

正式分析时，需要将以下上下文带入：

- 已确认的业务背景
- 已确认的粒度/口径
- 未解决异常列表
- 已忽略异常列表
- 用户自定义处理规则
- 图表判定结果

如果任何正式分析结论依赖被忽略异常，必须明确写出风险。

### Phase 5: Reconciliation Loop

当用户对当前结论不满意时，先问：

- 是业务背景理解不对
- 还是字段/口径判断不对
- 还是异常处理方式不对
- 还是分析范围、分组方式、时间窗口不对

只允许回退到下列检查点：

- `A`：分析目标与业务背景
- `B`：粒度 / 字段 / 指标口径
- `C`：异常处理决策
- `D`：当前 run 结果

### Phase 6: Finalize

最终报告前必须再做一次摘要确认：

- 当前结论是否基于可接受前提
- 是否接受已忽略异常带来的风险
- 是否还需要继续回退
- 报告导出格式是否仍按已确认设置

## 3. 输出风格

- 中文默认。
- 对管理层：结论先行。
- 对分析师：补充证据链和不确定性。
- 对异常：优先展示异常卡片，不要先灌长报告。
